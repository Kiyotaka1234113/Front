### Lab 9.1: Memoization with React.memo, useMemo, useCallback

## 1. Информация об использовании AI 
- **Инструмент:** Google Gemini
- **Дата:** 2026-03-14

## 2. Конкретные промпты

### Промпт 1:
"Как правильно продемонстрировать неоптимизированный Dashboard для Task 1, чтобы в консоли были видны лишние рендеры всех дочерних компонентов?"

**Что получил:** Разъяснение о том, что для первого задания нужно убрать все обертки `memo`, `useMemo` и `useCallback`, чтобы показать проблему «водопада рендеров» при каждом изменении стейта счетчика.

### Промпт 2:
"В чем разница между useMemo и useCallback, и в каких случаях их применять согласно лучшим практикам React?"

**Что получил:** Четкое разделение: `useMemo` используется для кеширования результатов тяжелых вычислений (например, цикла на 10 млн итераций), а `useCallback` — для сохранения ссылки на функцию между рендерами, чтобы не ломать мемоизацию дочерних компонентов.

### Промпт 3:
"Как типизировать пропсы для компонента, обернутого в React.memo с использованием TypeScript?"

**Что получил:** Примеры использования `interface` и правильного синтаксиса `export const Component = memo(function Component({ prop }: Props) { ... })`.

## 3. Модификация и проверка
- **Структура:** Сначала создал полностью неоптимизированную версию, проверил логи в консоли. Затем последовательно применил `memo` для `UserCard`, `useMemo` для `AnalyticsChart` и `useCallback` для кнопок.
- **Проверка:** Использовал `console.log` внутри компонентов и React DevTools Profiler. Убедился, что при клике на "Increment" перерисовывается только счетчик, а тяжелые вычисления и карточка пользователя игнорируются.
- **Чистота кода:** Вынес интерфейсы в отдельные типы и обеспечил строгую проверку TypeScript.

## 4. Что узнал
- Понял, как ссылочная идентичность (reference equality) объектов и функций в JavaScript влияет на процесс согласования (reconciliation) в React.
- Узнал правило "Don't optimize what doesn't need it" — мемоизация простых компонентов может наоборот замедлить приложение из-за накладных расходов на сравнение пропсов.
- Научился профилировать рендеры и находить узкие места.

### Lab 9.2: Virtualization for Large Lists

## 1. Информация об использовании AI
- **Инструмент:** Google Gemini
- **Дата:** 2026-03-14

## 2. Конкретные промпты

### Промпт 1:
"Как сгенерировать массив из 10 000 моковых объектов со случайными категориями и датами для тестирования виртуализации?"

**Что получил:** Функцию `generateItems`, использующую цикл `for` и `Math.random()` для создания массива типизированных объектов с нужными полями.

### Промпт 2:
"Uncaught TypeError: can't convert undefined to object react-window.js:457. Почему появляется эта ошибка при рендере FixedSizeList?"

**Что получил:** Узнал, что ошибка возникает из-за неправильной передачи пропса `style` в дочернюю функцию `FixedSizeList`. AI помог исправить деструктуризацию `{({ index, style }) => ...}` и указал на необходимость передачи `style` в корневой `div` элемента списка.

### Промпт 3:
"Uncaught SyntaxError: The requested module doesn't provide an export named: 'FixedSizeList'. Как исправить ошибку импорта в Vite?"

**Что получил:** Инструкцию по очистке кеша зависимости Vite (`rm -rf node_modules/.vite`) и альтернативный синтаксис импорта `import * as ReactWindow from "react-window"`.

## 3. Модификация и проверка
- **Сравнение производительности:** Создал компонент `RegularList` (без виртуализации) и `VirtualList` (с `react-window`). 
- **Интеграция поиска:** Реализовал фильтрацию списка через `useMemo`, чтобы поиск по 10 000 элементов не блокировал интерфейс.
- **Отладка:** Исправил проблемы с `undefined` стилями и очистил кэш сборщика Vite для корректной работы сторонней библиотеки.
- **Замеры:** Использовал Chrome DevTools (Performance tab) для сравнения времени рендера и количества созданных DOM-узлов (сокращение с 10000+ до ~15 узлов).

## 4. Что узнал
- Изучил концепцию "windowing" (виртуализации) — рендер только тех элементов, которые в данный момент находятся в видимой области экрана (viewport).
- Понял, как огромное количество DOM-узлов негативно влияет на First Contentful Paint (FCP) и плавность скроллинга.
- Узнал нюансы работы со сторонними библиотеками в Vite (очистка кэша сборки) и важность пропса `style` для абсолютного позиционирования элементов внутри виртуального контейнера.
